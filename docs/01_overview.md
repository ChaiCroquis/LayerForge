# 01. LayerForge - Project Overview

> Status: Design Phase
> Version: 0.4 (Claude Code skill 形態を第一実装として確定 - ADR-014)
> Date: 2026-05-11

---

## 1分要約

**LayerForge** は、自然言語情報を「**4±1レイヤー**に自動分解 → **各レイヤー内で蒸留** → **自然な文章として出力**」するシステム。

**中心原理**: **最小コンテキスト × ハルシネーション最小化** (ADR-012)
**実装原則**: 境界で推論、内部で決定論 (Inference at boundaries, determinism inside) (ADR-002)

---

## 中心原理の説明

LayerForge の全設計判断は、以下の原理階層から導出される：

```
目的: AI 応答の精度最大化
↓
手段: ハルシネーションの最小化
↓
方法: 最小コンテキストで AI を呼ぶ
↓
そのために: AI に渡す前に決定論で削れるものを削る
```

なぜ最小コンテキストがハルシネーション最小化につながるか：
- AI は入力が多いほど「関連付け」を増やそうとし、結果として捏造する
- 入力を本質だけに絞れば、AI の推論の自由度が減り、創作の余地が減る
- 「AI に最大情報を渡せば最良の答え」という直感は**逆**

**実装上の帰結**:
- 推論層は両端 (parse / render) のみ。中間は全て決定論
- 決定論層は pytest で 100% 検証可能
- プロンプトにペルソナ定義を使わない (ADR-011)
- schema で AI の出力範囲を厳しく制限する (ADR-007)

## 構想の起源

開発者 (本プロジェクト発案者) の構想：

1. 情報の意味的関係性を「重力」のメタファーで定式化
2. 同じ「重力係数」を持つノード集合を1つのレイヤーとして抽出
3. レイヤー数が **4±1** に収まるように重力係数を自動調整
4. 各レイヤー内では `(basis, law, residuals)` の形で蒸留
5. 最後に AI で自然な文章に翻訳して出力

この構想の動機階層：
- **最上位**: 楽して最大の効果を得たい
- **中位**: AIと決定論で分けたい (目的論=最小コンテキスト原理)
- **品質要件**: 機械検証可能・100%再現可能

---

## なぜ「成り立つ」のか

調査の結果、構想の各要素は既存研究で個別に実証済み：

| 要素 | 既存研究 | LayerForge での扱い |
|---|---|---|
| 階層クラスタリング | HERCULES (2025) | 縦軸の骨格として採用 (A1) |
| basis + 残差分解 | SCA (2024) | 横軸の蒸留式として採用 (A2) |
| 4±1 制約 | Cowan (2001) | 制約定数として採用 (A3) |
| 分離品質測定 | Newman modularity (2006) | 品質指標として採用 (A4) |
| RG的スケール変換 | RGMem, RG-DL (2026) | 内包確認（直接採用なし） |

**独自貢献**: 上記要素の統合と、**4±1 を「スケール係数を自動チューニングする観測指標として使う」発想**。既存研究はクラスタ数を手動 or 内部メトリクスで決めるが、4±1 を制御目標にする先行研究は確認できなかった。

## システム全体像

```
自然言語入力
   ↓ [INFERENCE BOUNDARY 1] AI推論 + schema強制
構造化データ (FormulationInput)
   ↓ [DETERMINISTIC CORE] 完全に決定論的
    ├─ 4±1 に収まるスケール係数を探索 (binary search)
    ├─ HERCULES adapted hierarchical clustering
    ├─ Newman modularity で品質チェック
    └─ SCA 蒸留 per layer
構造化結果 (CoreResult)
   ↓ [INFERENCE BOUNDARY 2] AI推論 + 翻訳のみ
自然言語出力 (レイヤー別の構造化テキスト)
```

推論層は **両端のみ**。中間処理は全てpytestで検証可能。

## 開発フェーズ計画

(ADR-010, ADR-013, ADR-014 参照)

**Phase 1: 公理層実装** (4-6週間)
- 採用論文の数式を pytest 化
- 決定論部分の全実装
- ※ 推論層 stub は不要 (skill 形態では Claude が直接担当)

**Phase 2a: SKILL.md ドラフト + Claude Code 統合** (**数日**) ← ADR-014で工数短縮
- SKILL.md の Mode A (自然言語整理) を記述
- CLI インターフェース (`layerforge.cli.decompose`) を実装
- 旧計画の "Claude API + structured output" は不要に

**Phase 2b: 認知補助具 (ドッグフーディング)** (2-3週間)
- SKILL.md の Mode B (決定整理) を記述
- CLI (`layerforge.cli.decide`) を実装
- 開発者自身のプロジェクト管理に使う
- 「閉じる/開く」判定ロジックを洗練
- LayerForge core を再利用、SKILL.md 指示のみ特化

**Phase 3: 実証実験** (期間未定)
- KDF ノード等で 4±1 収束率を測定
- 既存研究との比較
- 認知補助具のドッグフーディング結果を実証データに
- 論文化

**総工数**: 旧計画 (6-12週間) → 新計画 (約5-9週間)

## 主要な技術判断（要約）

詳細は ADR を参照。

| Decision | 内容 | ADR |
|---|---|---|
| 推論と決定論の分離 | 全フローに「境界で推論」原則を適用 | 002 |
| 4±1 = [3, 5] | レイヤー数の制約範囲 | 003 |
| 重力 → スケール係数 | 比喩は維持、実装は中立用語 | 004 |
| 採用論文絞り込み | 内包確認できる派生は採用しない | 005 |
| 公理テスト | 数式を境界条件としてテスト化 | 006 |
| schema 駆動 | frozen dataclass で境界の型を固定 | 007 |
| 2段階フォールバック | retry → deterministic alternative | 008 |
| 独立サブシステム | KDF/AET-OS/TIVE とは後段で統合 | 009 |
| 3フェーズ開発 | 公理層 → 推論接続 → 実証 | 010 |
| ペルソナ定義禁止 | プロンプトで "You are a..." 形式を使わない | 011 |
| 最小コンテキスト原理 | 全設計判断の上位原理として明示 | 012 |
| 認知補助具を最初の応用 | 決定整理（ドッグフーディング）を Phase 2b に追加 | 013 |
| **skill 形態を第一実装** | Claude Code skill として実装、API 不要 | **014** |

## 期待される成果

### 短期 (Phase 1-2a 完了時)

- 自然言語入力に対して 4±1 レイヤーへの自動分解が動作
- pytest 150-200 本で全公理が保証された決定論コア
- LLM 障害時にもフォールバックで出力継続

### 中短期 (Phase 2b 完了時) ← **新規 (ADR-013)**

- **認知補助具 (CLI) が稼働**: タスク入力 → 「今決めるべき決定群 (4±1個)」を出力
- 開発者自身がプロジェクト管理に日常使用
- ドッグフーディングによる設計改善のフィードバックループ確立

### 中期 (Phase 3)

- KDF統合により、知識ノードの自動階層化
- TIVE 280原則をテストケースとして再利用可能
- 論文化（既存研究の統合点 + 4±1 観測指標の独自性 + 認知補助具の実証例）

### 長期 (Phase 3 以降)

- AET-OS の思考整理エージェントとして組み込み
- Claude Code の skill / hooks 化
- 認知科学・物理・NLPを横断する統合理論の実装例として位置づけ

## リスクと対処

| リスク | 影響 | 対処 |
|---|---|---|
| 4±1 への収束が稀にしか起きない | 構想の核心が破綻 | Phase 1 完了時に実証実験、調整可能パラメータの追加 |
| LLM API の不安定性 | 推論層が止まる | フォールバック実装で最低限の出力保証 |
| 計算コスト過大 | 実用困難 | Phase 1 で n=10,000 規模のベンチマーク |
| 既存研究との差分が薄い | 論文化困難 | 4±1 観測指標の独自性を強調、実証実験で差を出す |
| 開発者本人の負荷増 | 並行プロジェクトが詰まる | 独立サブシステムとして開発、統合は後段 (ADR-009) |

## 参考: 開発者構想の元の発言（時系列要約）

1. 「情報の密度って計算的に出来たりするんだろうか？」→ embedding ベースの密度測定
2. 「意味的関係性だな」→ 関係性に焦点
3. 「重力の影響度合いがわかるかなと」→ 重力アナロジー導入
4. 「同じレイヤーとは、同じ計算式が成り立つレイヤー」→ レイヤー定義
5. 「ワーキングメモリ、5～7,8の部分」→ 4±1 に更新
6. 「4±1の範囲より多い場合、粒度が荒いか細かい」→ **観測指標としての 4±1**（独自貢献の核心）
7. 「決定論として記述可能なものとAIとして推論しないと出来ないものを分けておきたい」→ ADR-002
8. 「蒸留結果をAIに投げて、自然な文章にしてもらう」→ Boundary 2 設計
9. 「同じ数式で情報の蒸留」→ SCA 採用
10. 「使える公式は使う、それに必要な情報を各論文から抽出」→ 本設計資料の作成方針

この10ステップの議論で、構想は実装可能なレベルまで具体化された。
