Python の新規プロジェクトに unit test を導入したいということで、ここでは pytest を中心に据えた現代的なアプローチをご紹介します。pytest は標準ライブラリの unittest よりも記述量が少なく、フィクスチャ機構やプラグインエコシステムが豊富で、新規プロジェクトの初期から採用する価値が十分あります。まずはプロジェクトルートに tests/ ディレクトリを作り、pyproject.toml に [tool.pytest.ini_options] セクションを設けて testpaths や addopts を指定するところから始めるとよいでしょう。最初の一歩としては、`pip install pytest pytest-cov` を行い、tests/test_smoke.py に `def test_truth(): assert 1 + 1 == 2` のような最小のテストを書いて、`pytest -q` が緑になる状態を確立することが重要です。

最初の最小テストが走るようになったら、次に意識すべきはテストの命名規約と発見ルールです。pytest はデフォルトで `test_*.py` または `*_test.py` のファイル名、`Test*` で始まるクラス、`test_*` で始まる関数を自動収集します。新規プロジェクトであればこのデフォルトに従うのが最も摩擦が少なく、後からチームメンバーが参加したときの認知コストも下がります。クラスを使う必要は基本的になく、モジュールレベルの関数で書くのが pytest 流のスタイルだと割り切ってよいでしょう。

カバレッジ計測は、テストを書く文化が定着するかどうかに直結する重要な装置です。pytest-cov をインストールし、`pytest --cov=src --cov-report=term-missing --cov-report=html` のように実行すれば、ターミナルに未到達行が表示され、htmlcov/ ディレクトリにブラウザで閲覧できるレポートが生成されます。新規プロジェクトでは最初から branch coverage(`--cov-branch`)を有効化しておくと、if 文の片側しか通っていないケースを早期に発見できます。ただし、カバレッジ率を KPI 化しすぎると「カバレッジのためのテスト」が増えて本来の検証価値が下がるため、80% 程度を目安にしつつ、未到達行のレビューを重視する運用が現実的です。

assertion の書き方は pytest の最大の強みのひとつで、`assert expected == actual` というプレーンな Python 文を書けば、失敗時には pytest が AST を書き換えて差分情報を整形表示してくれます。unittest の `self.assertEqual` のような専用 API を覚える必要はなく、`assert "foo" in result`、`assert len(items) == 3`、`assert isinstance(x, MyClass)` のように自然な Python 表現でほぼ全ての検証を書けます。複雑なオブジェクトを比較するときは、まず辞書化してから比較する、あるいは dataclass の `__eq__` を活用するのが定石です。

CI/CD への接続は、テストを「書いた」状態から「常に動いている」状態へ昇格させる肝です。GitHub Actions であれば、`.github/workflows/ci.yml` に Python のセットアップステップ、依存インストール、`pytest --cov --junitxml=junit.xml` の実行、そして coverage と junit のアーティファクトアップロードを並べるのが標準構成です。matrix.python-version に `["3.10", "3.11", "3.12"]` を並べれば複数の Python バージョンで横断的に検証でき、OS matrix(ubuntu-latest, macos-latest, windows-latest)まで広げるとプラットフォーム依存のバグも早期に検出できます。

フィクスチャは pytest のもうひとつの主役で、`@pytest.fixture` デコレータで定義した関数の戻り値をテスト関数の引数として受け取れる仕組みです。`conftest.py` にフィクスチャを置けばディレクトリ配下のすべてのテストから自動的に利用でき、import 文は不要です。スコープは function(デフォルト)、class、module、session の 4 段階があり、DB コネクションのような重い資源は session スコープに、テストごとにリセットしたい状態は function スコープに割り当てるのが基本パターンです。

ドキュメントを忘れがちですが、新規プロジェクトでは README.md の早い段階に「テストの走らせ方」セクションを設けることを強くおすすめします。`pytest`、`pytest -k test_login`、`pytest -x --pff`、`pytest --cov` の 4 つくらいのコマンド例を載せておけば、初参加のメンバーが 1 分で動かせるようになります。さらに各テストファイルの冒頭にモジュール docstring を書き、「何の振る舞いを検証するためのテスト群か」を 1〜2 行で説明する習慣をつけると、半年後の自分が救われます。

テストデータの管理は、プロジェクトが大きくなるにつれて重要度が増す論点です。小さな入力値であればテスト関数内にリテラルで書くのが最も明快ですが、複雑なオブジェクトを複数のテストで使い回す場合はフィクスチャに切り出します。さらに、エンティティのバリエーションを増やしたいときには、`factory_boy` や `pydantic` ベースの自作 factory を導入すると、デフォルト値を持つビルダー的な API でテストデータを構築でき、「id だけ変えたい」「email だけ無効値にしたい」といったケースの記述量が劇的に減ります。

エッジケースの設計は、新規プロジェクトのテスト文化を成熟させる試金石です。空入力(空リスト、空文字、None)、境界値(0、1、最大値、最大値+1)、型エラー(数値を期待する箇所に文字列)、ロケール依存(タイムゾーン、文字コード、小数点記号)の 4 カテゴリは、テストを書くたびに「この 4 つは確認したか?」というチェックリストとして頭に置くとよいでしょう。多くの本番障害は、Happy path のテストでは届かないこれらの隅で発生します。

パラメトリックテストは pytest の `@pytest.mark.parametrize` デコレータで実現でき、ひとつのテスト関数に複数の入力と期待値の組を渡すことができます。`@pytest.mark.parametrize("n,expected", [(0, 1), (1, 1), (5, 120)])` のように書けば、それぞれの組について独立したテストケースとして実行され、失敗時にはどの組で落ちたかが一目でわかります。エッジケースの網羅と組み合わせると、テーブル駆動テストとして非常に強力です。

PR テンプレートは、`.github/PULL_REQUEST_TEMPLATE.md` を置くことで GitHub が自動的に PR 本文にプレースホルダーとして展開してくれます。新規プロジェクトでは「変更概要」「テスト方針」「カバレッジ影響」「破壊的変更の有無」「レビュアーへの注意事項」の 5 項目程度に絞ったテンプレートにしておくと、書き手の負担が小さく、かつレビュアーが評価軸を見失わない構造になります。テストに関する項目を必ず含めることで、PR 作成者に「このコードはどう検証されているか」を毎回言語化させる効果が生まれます。

ゴールデンファイル(snapshot)テストは、JSON や HTML のような構造化テキストの出力を検証するときに威力を発揮します。`syrupy` プラグインを使えば、初回実行時に出力をスナップショットとして保存し、次回以降は保存済みスナップショットと比較する形で検証できます。出力フォーマットが安定してきた段階で導入すると、リグレッション検出器として極めて効率的に機能しますが、出力が頻繁に変わるフェーズで導入するとスナップショット更新が形骸化するので、タイミングは見極めが必要です。

ベンチマークについては `pytest-benchmark` を導入しておくと、`def test_speed(benchmark): benchmark(my_func, arg)` のように書くだけで、関数の実行時間統計(min、max、mean、stddev)を自動的に収集してくれます。CI で結果を JSON に保存し、過去の値と比較してしきい値を超えたら fail させるという運用にすれば、コミットごとの性能リグレッションを検出する装置になります。ただし、CI ランナーは性能が安定しないので、絶対値ではなく相対比較を中心に据えるのが現実的です。

CI のアーティファクト機能は意外と活用されていない宝物で、`actions/upload-artifact@v4` を使ってカバレッジ HTML、junit XML、スクリーンショット、ログファイルなどを CI ジョブから保存できます。失敗した PR のログをいちいち CI 画面で探す手間が省け、レビュアーがローカルで再現する前にアーティファクトを確認するというフローが定着すると、レビュー回転速度が体感で 2〜3 割向上します。新規プロジェクトでは最初からこの仕組みを組み込んでおくと、後から導入するコストを回避できます。

mock の使い方は、テストを書く文化が崩壊する第一の原因になりがちな分野です。`unittest.mock.patch` や `pytest-mock` の `mocker` フィクスチャは強力ですが、内部実装に強く結合したテストを書きやすく、リファクタリング時に大量のテストが赤くなる悪夢を招きます。原則として、外部 I/O(HTTP、DB、ファイルシステム、時刻)の境界だけ mock し、純粋なビジネスロジックは mock せずに実物を組み合わせてテストするのが、長期的にメンテナンス可能なテストになります。

プロパティベーステストは、`hypothesis` ライブラリを導入することで実現できます。`@given(st.integers())` のようにデコレータで入力ドメインを宣言すると、Hypothesis が自動的にエッジケースを含む多数の入力でテストを実行し、失敗時には最小化された反例を提示してくれます。文字列処理、数値計算、シリアライズ/デシリアライズの round-trip 性質などには劇的に効きます。新規プロジェクトでも、コア関数だけは hypothesis でテストする習慣をつけておくと、隠れたバグの発見率が大きく向上します。

レビュアーチェックリストは、PR テンプレートとセットで運用すると効果が増幅します。「テストは追加されているか」「エッジケースのテストはあるか」「カバレッジは下がっていないか」「テスト名は意図を表現しているか」「mock は外部境界に限定されているか」といった項目を 5〜7 個に絞ったチェックリストを CONTRIBUTING.md に書いておくと、レビュアー間で品質基準のばらつきが減ります。チェックリスト自体は短くすることが重要で、20 項目もあると誰も読まなくなります。

ブロッキング条件、つまり「これに該当したら必ず merge を止める」という基準は、新規プロジェクトの早い段階で明文化しておくと後の摩擦を減らせます。最低限の例としては「CI が赤」「カバレッジが N% 下がる」「破壊的変更にバージョン bump がない」「セキュリティ警告(dependabot、bandit)が新規発生」の 4 つあたりが王道です。これらを GitHub の branch protection rule や required checks として物理層に落とし込むと、「気をつける」依存ではなく仕組みで担保されます。

docstring とテストの関係は、`doctest` モジュールでひとつ示唆深いつながりを持ちます。pytest は `--doctest-modules` オプションで docstring 内の `>>>` 例を自動的にテストとして実行してくれるため、ドキュメントに書いたサンプルコードが常に動作することを保証できます。API ドキュメントとテストが分離して片方だけ古くなる問題を構造的に防げるので、ライブラリ系プロジェクトでは積極的に活用したい仕組みです。

CI 上の matrix build をうまく使うコツは、「すべての組み合わせを総当たり」ではなく「重要な軸だけクロスする」ことです。Python 3.10/3.11/3.12 × Ubuntu/macOS/Windows = 9 通りの組み合わせは、Windows × macOS 環境を持たない開発者にとっては事実上の唯一の検証手段なので、新規プロジェクトでも最初から組んでおく価値があります。一方、依存パッケージのバージョン matrix まで広げると CI 時間が膨らみすぎるので、別ワークフロー(nightly run)に切り出すのが定石です。

境界値テストの具体例として、年齢を扱う関数なら 0、-1、120、121、None、"5"(文字列)、5.5(浮動小数)を全部投げ込むのが基本セットです。リストを扱う関数なら []、[None]、[同一要素 × 1万]、[巨大文字列を含む]、ジェネレータ(リストではない iterable)などを試します。文字列処理なら ""、" "、"\n"、"あ"(マルチバイト)、"👨‍👩‍👧"(grapheme cluster)、"a" × 10**6(巨大入力)が定番です。これらを「最初に書くテンプレート」として頭に置いておくと、自然とエッジケースが拾えるようになります。

新規メンバー向けのオンボーディング文書には、「失敗するテストの読み方」のセクションを設けることを強くおすすめします。pytest の assertion 失敗出力は情報量が多い反面、初学者には読みづらく、`E       assert 3 == 4` の上下に表示される文脈をどう辿るかを 1 ページにまとめておくだけで、つまずきが激減します。さらに、`pytest -vv --tb=long --showlocals` のような診断用コマンドも併記しておくと親切です。

性能テストのリグレッション検出は、絶対閾値(「100ms 以内」)よりもベースライン比較(「過去 30 日の中央値の 1.3 倍以内」)の方が、CI ランナーのばらつきに強い指標になります。`pytest-benchmark` には `--benchmark-compare` オプションがあり、過去の結果と比較できます。これを `--benchmark-compare-fail=mean:10%` のように設定すると、平均が 10% 以上劣化したらテストが fail する形に組めます。アラートは Slack や PR コメントに流す運用にすると、性能劣化を見逃しにくくなります。

ロケール依存の罠は、特に日本語環境の Python で顕在化しがちです。datetime のフォーマットが locale.LC_TIME に依存する、`str.lower()` がトルコ語ロケールで予想外の挙動をする、ファイル名の正規化(NFC/NFD)が macOS と Linux で違う、といった現実があります。テストでこれらを検出するには、CI matrix に `TZ=Asia/Tokyo` と `TZ=UTC` の両方を含める、`LANG=ja_JP.UTF-8` と `LANG=C` の両方で走らせる、といった工夫が有効です。

新規プロジェクトでフィクスチャと factory を併用する場合の使い分けは、「コードベース全体で 1 種類しか使わないもの(DB セッション、設定オブジェクト)はフィクスチャ、複数のバリエーションを生成するもの(ユーザー、注文、商品)は factory」という線引きが現実的です。両者を混在させても問題はなく、factory が返したオブジェクトをフィクスチャでセットアップする組み合わせも頻出パターンです。

profile を取りたい場面では、`pytest --profile` を提供する `pytest-profiling` プラグインや、pyinstrument を組み合わせる方法があります。CI で常に取るのは過剰ですが、性能リグレッションが発生したときに手元で `pytest tests/test_slow.py --profile-svg` を叩けば、ボトルネックの呼び出しグラフが SVG として得られます。プロファイルとベンチマークは別の道具で、ベンチマークが「どれくらい遅いか」を答え、プロファイルが「なぜ遅いか」を答えるという役割分担を意識すると混乱しません。

git のフックを活用するなら、`pre-commit` フレームワークを導入して、コミット前に `pytest tests/test_fast.py`(高速サブセット)、ruff、mypy などを走らせる構成が王道です。pre-commit hook はチーム全員のローカル環境で同じ品質ゲートを実現できるので、CI に到達する前に大半の落ちを潰せます。新規プロジェクトでは最初から `.pre-commit-config.yaml` を整備しておくと、後から導入するときに発生する「すでに違反している既存コード」問題を回避できます。

テスト名は、`test_<対象>_<条件>_<期待>` のように構造化すると、失敗時に何が壊れたかが名前だけで把握できます。例えば `test_login_with_invalid_password_returns_401` のように長くても、失敗時の認知コストを下げる効果は大きいです。日本語コメントで補足するのもよく、`def test_長文URL():  # 2083 文字超の URL を渡しても切り捨てずに保存される` のように、関数名と docstring/コメントで意図を二重に明示すると、半年後に読み返したときの理解が早まります。

カバレッジレポートを CI に統合するときは、`codecov` や `coveralls` のような外部サービスに連携するか、PR コメントとしてカバレッジ差分を投稿する自作 Action を組むかの 2 択になります。新規プロジェクトでは codecov の方が導入が早く、PR ごとに「この PR でカバレッジが N% 増加/減少した」という diff コメントが自動で付くので、レビュアーが意識せずともカバレッジ状況を把握できます。

例外を期待するテストは、`pytest.raises` コンテキストマネージャで書くのが標準形です。`with pytest.raises(ValueError, match="invalid"): do_something()` のように、例外型だけでなくメッセージのパターンも検証すると、「意図しない別の理由で同じ型の例外が出た」という偽陽性を防げます。さらに `excinfo.value.args` を後段で参照すれば、例外オブジェクトの属性まで検証可能です。

非同期コードのテストには `pytest-asyncio` プラグインが定番で、`@pytest.mark.asyncio` を付けた async 関数として書けます。`asyncio` だけでなく `anyio` や `trio` を使う場合は `anyio` プラグインが選択肢になります。新規プロジェクトで async を採用するなら、最初のテスト 1 件目から非同期前提で書き始めると、後から sync/async 混在で苦しむことが減ります。

成熟段階に応じて、テストレイヤーの構造化を意識するとよいでしょう。「ユニットテスト(関数/クラス単位)」「インテグレーションテスト(複数モジュール組み合わせ)」「E2E テスト(外部から見た振る舞い)」の 3 層に分けて tests/unit/、tests/integration/、tests/e2e/ ディレクトリに置き、pytest の marker(`@pytest.mark.slow` など)で制御するのが定石です。新規プロジェクトでは最初は unit だけでよく、必要になった段階で他の層を増やせばよいです。

スナップショットテストの運用上の注意は、「スナップショット更新が儀式化しないようにする」ことです。失敗したら無条件に `--snapshot-update` を叩く文化が定着すると、せっかくの検出力が形骸化します。PR でスナップショットファイルが変更されているときは、レビュアーが必ず「この変更は意図通りか」を確認する、というルールをチェックリストに入れておくと健全に運用できます。

ドキュメント文化の一環として、各テストファイルの冒頭に「このテストが守っている契約」を 3 行程度で書く習慣をつけると、テスト自体が仕様書として機能します。「このモジュールの login() 関数は、有効な認証情報で User オブジェクトを返し、無効な情報で AuthError を投げ、過剰なリトライで RateLimitError を投げる」のような契約宣言は、テストを書くときの設計指針にもなり、レビュー時の評価軸にもなります。

CI のキャッシュ設定も、テスト体験に直接効きます。`actions/cache@v4` で pip キャッシュ、uv/poetry のキャッシュ、`.pytest_cache` をキャッシュしておくと、テスト実行までの待ち時間が大きく短縮されます。特に Windows ランナーは依存インストールが遅いので、キャッシュの効果が顕著です。新規プロジェクトでも、最初の workflow ファイルに最初からキャッシュステップを入れておくと損はありません。

レビューチェックリストに「テストの可読性」を入れるかどうかは判断が分かれる論点ですが、新規プロジェクトでは入れることをおすすめします。「テスト関数が 30 行を超えていないか」「Arrange-Act-Assert の構造が明瞭か」「マジックナンバーに名前が付いているか」のような項目は、本体コードの可読性レビューと同じ基準を適用するという宣言になり、テストコードを「二級市民」として扱わない文化を作れます。

ゴールデンファイルを保存するディレクトリは、tests/__snapshots__/ や tests/fixtures/golden/ のような専用ディレクトリにまとめ、Git LFS に乗せるかどうかは出力サイズに応じて判断します。バイナリ出力(画像、PDF)を比較する場合は、ピクセル単位の完全一致ではなく許容差を持った diff 関数(`numpy.allclose`、`pixelmatch`)を経由するのが現実的です。

新規プロジェクトでよくある罠として、「テストファイルから本体コードへの import が動かない」問題があります。これは `pyproject.toml` の `[tool.pytest.ini_options]` で `pythonpath = ["src"]` を指定するか、本体パッケージを editable install(`pip install -e .`)するかで解決します。新規プロジェクトの初期セットアップ時に、この設定だけはチームメンバーがハマる前に整備しておくと親切です。

最後に、テスト導入の心構えとして、「テストは資産であると同時に負債でもある」という認識を持つとバランスが取れます。網羅性を上げすぎると変更コストが膨らみ、緩めすぎるとリグレッション検出力が下がるという二律背反の中で、プロジェクトのフェーズ(プロトタイプ/MVP/本番)に応じて密度を調整するのが現実的です。新規プロジェクトの初期はカバレッジ 60% 程度、コア機能のみエッジケース網羅、外部 I/O は最小限の mock、という構成から始めて、機能が安定するに連れて徐々に厳格化していく、というロードマップが多くのプロジェクトで機能します。これらの仕組みを最初の 1 週間で 7〜8 割整えてしまえば、半年後のあなた自身と、その頃に参加するであろう新メンバーの双方にとって、生産性の高い開発環境が出来上がっているはずです。
