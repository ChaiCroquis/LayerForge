## CI/CD

新規 Python プロジェクトでは、最初のコミット時点から CI を整備しておくことを強く推奨します。GitHub Actions を採用する場合、`.github/workflows/ci.yml` に Python のセットアップ、依存インストール、テスト実行という 3 ステップを並べる構成が標準です。

matrix build は、複数の Python バージョンと OS を横断的に検証する強力な仕組みです。`matrix.python-version` に `["3.10", "3.11", "3.12"]` を並べ、さらに OS を `ubuntu-latest`, `macos-latest`, `windows-latest` まで広げると、プラットフォーム依存のバグを早期に検出できます。Windows / macOS 環境を持たない開発者にとっては、CI が事実上の唯一の検証手段になるため、新規プロジェクトでも最初から組んでおく価値があります。一方、依存パッケージのバージョン matrix まで広げると CI 時間が膨らみすぎるので、別ワークフロー (nightly run) に切り出すのが定石です。

ロケール依存の罠を防ぐため、CI matrix に `TZ=Asia/Tokyo` と `TZ=UTC`、`LANG=ja_JP.UTF-8` と `LANG=C` の両方を含める工夫も有効です。

CI のアーティファクト機能は活用余地が大きく、`actions/upload-artifact@v4` を使って junit XML、スクリーンショット、ログファイルなどを保存できます。失敗ジョブの調査効率が体感で 2〜3 割向上します。

ブロッキング条件は早い段階で明文化しておきましょう。「CI が赤」「破壊的変更にバージョン bump がない」「セキュリティ警告 (dependabot、bandit) が新規発生」などを GitHub の branch protection rule や required checks として物理層に落とし込めば、「気をつける」依存ではなく仕組みで品質を担保できます。
