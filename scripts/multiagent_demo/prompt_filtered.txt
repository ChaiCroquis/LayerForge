# AI 1 の回答 (LayerForge で CI/CD layer のみ抽出済)

CI/CD への接続は、テストを「書いた」状態から「常に動いている」状態へ昇格させる肝です。GitHub Actions であれば、`.github/workflows/ci.yml` に Python のセットアップステップ、依存インストール、`pytest --cov --junitxml=junit.xml` の実行、そして coverage と junit のアーティファクトアップロードを並べるのが標準構成です。matrix.python-version に `["3.10", "3.11", "3.12"]` を並べれば複数の Python バージョンで横断的に検証でき、OS matrix(ubuntu-latest, macos-latest, windows-latest)まで広げるとプラットフォーム依存のバグも早期に検出できます。

PR テンプレートは、`.github/PULL_REQUEST_TEMPLATE.md` を置くことで GitHub が自動的に PR 本文にプレースホルダーとして展開してくれます。新規プロジェクトでは「変更概要」「テスト方針」「カバレッジ影響」「破壊的変更の有無」「レビュアーへの注意事項」の 5 項目程度に絞ったテンプレートにしておくと、書き手の負担が小さく、かつレビュアーが評価軸を見失わない構造になります。テストに関する項目を必ず含めることで、PR 作成者に「このコードはどう検証されているか」を毎回言語化させる効果が生まれます。

ベンチマークについては `pytest-benchmark` を導入しておくと、`def test_speed(benchmark): benchmark(my_func, arg)` のように書くだけで、関数の実行時間統計(min、max、mean、stddev)を自動的に収集してくれます。CI で結果を JSON に保存し、過去の値と比較してしきい値を超えたら fail させるという運用にすれば、コミットごとの性能リグレッションを検出する装置になります。ただし、CI ランナーは性能が安定しないので、絶対値ではなく相対比較を中心に据えるのが現実的です。

CI のアーティファクト機能は意外と活用されていない宝物で、`actions/upload-artifact@v4` を使ってカバレッジ HTML、junit XML、スクリーンショット、ログファイルなどを CI ジョブから保存できます。失敗した PR のログをいちいち CI 画面で探す手間が省け、レビュアーがローカルで再現する前にアーティファクトを確認するというフローが定着すると、レビュー回転速度が体感で 2〜3 割向上します。新規プロジェクトでは最初からこの仕組みを組み込んでおくと、後から導入するコストを回避できます。

ブロッキング条件、つまり「これに該当したら必ず merge を止める」という基準は、新規プロジェクトの早い段階で明文化しておくと後の摩擦を減らせます。最低限の例としては「CI が赤」「カバレッジが N% 下がる」「破壊的変更にバージョン bump がない」「セキュリティ警告(dependabot、bandit)が新規発生」の 4 つあたりが王道です。これらを GitHub の branch protection rule や required checks として物理層に落とし込むと、「気をつける」依存ではなく仕組みで担保されます。

CI 上の matrix build をうまく使うコツは、「すべての組み合わせを総当たり」ではなく「重要な軸だけクロスする」ことです。Python 3.10/3.11/3.12 × Ubuntu/macOS/Windows = 9 通りの組み合わせは、Windows × macOS 環境を持たない開発者にとっては事実上の唯一の検証手段なので、新規プロジェクトでも最初から組んでおく価値があります。一方、依存パッケージのバージョン matrix まで広げると CI 時間が膨らみすぎるので、別ワークフロー(nightly run)に切り出すのが定石です。

ロケール依存の罠は、特に日本語環境の Python で顕在化しがちです。datetime のフォーマットが locale.LC_TIME に依存する、`str.lower()` がトルコ語ロケールで予想外の挙動をする、ファイル名の正規化(NFC/NFD)が macOS と Linux で違う、といった現実があります。テストでこれらを検出するには、CI matrix に `TZ=Asia/Tokyo` と `TZ=UTC` の両方を含める、`LANG=ja_JP.UTF-8` と `LANG=C` の両方で走らせる、といった工夫が有効です。

新規プロジェクトでフィクスチャと factory を併用する場合の使い分けは、「コードベース全体で 1 種類しか使わないもの(DB セッション、設定オブジェクト)はフィクスチャ、複数のバリエーションを生成するもの(ユーザー、注文、商品)は factory」という線引きが現実的です。両者を混在させても問題はなく、factory が返したオブジェクトをフィクスチャでセットアップする組み合わせも頻出パターンです。

profile を取りたい場面では、`pytest --profile` を提供する `pytest-profiling` プラグインや、pyinstrument を組み合わせる方法があります。CI で常に取るのは過剰ですが、性能リグレッションが発生したときに手元で `pytest tests/test_slow.py --profile-svg` を叩けば、ボトルネックの呼び出しグラフが SVG として得られます。プロファイルとベンチマークは別の道具で、ベンチマークが「どれくらい遅いか」を答え、プロファイルが「なぜ遅いか」を答えるという役割分担を意識すると混乱しません。

git のフックを活用するなら、`pre-commit` フレームワークを導入して、コミット前に `pytest tests/test_fast.py`(高速サブセット)、ruff、mypy などを走らせる構成が王道です。pre-commit hook はチーム全員のローカル環境で同じ品質ゲートを実現できるので、CI に到達する前に大半の落ちを潰せます。新規プロジェクトでは最初から `.pre-commit-config.yaml` を整備しておくと、後から導入するときに発生する「すでに違反している既存コード」問題を回避できます。

カバレッジレポートを CI に統合するときは、`codecov` や `coveralls` のような外部サービスに連携するか、PR コメントとしてカバレッジ差分を投稿する自作 Action を組むかの 2 択になります。新規プロジェクトでは codecov の方が導入が早く、PR ごとに「この PR でカバレッジが N% 増加/減少した」という diff コメントが自動で付くので、レビュアーが意識せずともカバレッジ状況を把握できます。

非同期コードのテストには `pytest-asyncio` プラグインが定番で、`@pytest.mark.asyncio` を付けた async 関数として書けます。`asyncio` だけでなく `anyio` や `trio` を使う場合は `anyio` プラグインが選択肢になります。新規プロジェクトで async を採用するなら、最初のテスト 1 件目から非同期前提で書き始めると、後から sync/async 混在で苦しむことが減ります。

新規プロジェクトでよくある罠として、「テストファイルから本体コードへの import が動かない」問題があります。これは `pyproject.toml` の `[tool.pytest.ini_options]` で `pythonpath = ["src"]` を指定するか、本体パッケージを editable install(`pip install -e .`)するかで解決します。新規プロジェクトの初期セットアップ時に、この設定だけはチームメンバーがハマる前に整備しておくと親切です。

  - テストデータ管理 / fixtures / factories
  - エッジケース / 境界値テスト
  - ドキュメント / docstring / README の他セクション
  - コードレビュー / PR テンプレート
  - パフォーマンステスト / benchmark
- 「上記の制約に従い」のような meta-注記は書かないこと
